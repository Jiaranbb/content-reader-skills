# Bilibili Reader JR 工作流

## 默认路径：只保存逐字稿

1. 识别输入链接类型。
   - 单视频：直接处理。
   - 分 P / 合集 / 播放列表：用 `yt-dlp --flat-playlist --dump-single-json` 展开条目，一条视频生成一篇笔记。
2. 获取元数据：标题、作者、发布时间、时长、规范链接。
3. 优先从用户主浏览器登录页面读取互动数据：
   - 播放、弹幕、点赞、投币、收藏、分享、评论。
   - 浏览器提取结果保存为 JSON 后传给脚本 `--browser-meta-json`。
   - 不导出 cookies、token、localStorage。
4. 优先使用用户主浏览器登录页面可见字幕：
   - 打开用户主浏览器里的 B 站视频页。
   - 如果页面字幕按钮可见且可以选择「中文」，先从浏览器页面提取字幕。
   - 浏览器提取结果保存为 JSON 后传给脚本 `--transcript-json`。
   - 不导出 cookies、token、localStorage。
5. 获取平台字幕：
   - 优先 `zh-CN,zh-Hans,zh,en`
   - 支持 VTT、SRT、B 站 JSON 字幕。
6. 清洗字幕：
   - 去掉时间戳、HTML 标签、重复片段。
   - 合并短句，保留可读段落。
7. 保留字幕时间戳，生成「媒体文字内容」。
8. 根据逐字稿生成「划重点」：
   - 每条必须包含时间戳。
   - 时间戳后面必须写“从这里开始看什么”，例如某个概念解释、某个案例、某个论证段落、某个操作建议。
   - 不要只是复制该时间点的一句字幕；要看该时间点后约 60-90 秒内容，提炼成可回看的观看导航。
   - 优先挑出问题定义、核心概念、推导转折、结论、案例、操作建议。
   - 目标是辅助回看，不做全文摘要，也不写空泛标签。
   - 自动生成后必须快速审阅；如果某条只是原字幕原句，改写成“看这里如何说明/解释/演示/比较/给出建议”。
9. 如果浏览器字幕和平台字幕都不可用，才下载临时音频并用 `faster-whisper` 本地转写。
10. 生成「内容摘要」：
   - 必须根据完整逐字稿提炼 3-4 条核心观点/知识点。
   - 每条尽量控制在 60 个中文字符以内。
   - 不要按段落复述，也不要复制「划重点」里的观看导航。
   - 不允许留空，也不允许只写 `待整理`。
   - 自动摘要只作为兜底；如果内容泛泛，执行 agent 必须读逐字稿后人工改写。
11. 写入 Obsidian Markdown 笔记。头部必须包含互动数据；缺失字段写 `-`。

## 显式媒体下载路径

- `--action download-media`：只下载视频，不写笔记。
- `--include-media`：先写逐字稿笔记，再下载视频，并在笔记中加入 Obsidian 可预览的 `<video>` 标签。
- 默认 `--action transcript` 不下载媒体；字幕阶段使用 `yt-dlp --skip-download`。

## 批量处理规则

- 用户明确说「全部」时可以处理整个分 P / 合集 / 播放列表。
- 用户只说「这个合集」但没有说明范围，且条目较多时，先询问范围或建议 `--max-items`。
- 每条视频独立写一篇 Markdown，避免合集笔记过长。

## 登录与隐私边界

- 默认不使用浏览器 cookies。
- 不输出、不保存 cookies、token、localStorage。
- 如果 B 站返回会员、登录或地区限制，直接报告阻塞原因，不绕过限制。
